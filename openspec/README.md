# OpenSpec: Specification-Driven Development

## Структура

```
openspec/
├── specs/          # Источник истины — спецификации возможностей
└── changes/        # Активные предложения изменений
```

## Workflow

1. **Draft** — создание предложения в `changes/[change-id]/proposal.md`
2. **Review** — обсуждение и доработка
3. **Implement** — реализация согласно спецификации
4. **Complete** — обновление спецификаций в `specs/`
5. **Archive** — перемещение завершённых изменений

## Формат спецификации

См. примеры в `specs/` и шаблоны в документации.

## Использование

- **Аналитик**: создаёт/обновляет спецификации в `specs/`, пишет предложения
- **Планировщик**: читает спецификации для требований, создаёт `tasks.md` в изменениях
- **Архитектор**: пишет `design.md` в предложениях изменений
- **Разработчик**: реализует согласно спецификациям, обновляет дельты спецификаций
- **Ревьюер**: проверяет соответствие требованиям спецификаций

## Интеграция с существующими specs

В проекте также есть `specs/capabilities/` (Spec Kit) и `changes/active/` (старая структура).
OpenSpec работает параллельно с ними, но рекомендуется мигрировать в `openspec/` для единообразия.
